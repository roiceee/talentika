# Performance and Scalability

To improve performance as the system scales to handle more applicants, several adjustments can be made. First, more worker containers can be added behind the same Redis queue, so résumés are processed in parallel by more workers at once. Second, the concurrency setting on each worker can be raised above three, but only on machines that have enough CPU cores to support it, since the OCR worker slows down when CPU is overloaded. Third, the OpenAI rate limit on the project account should be raised before adding more AI workers, because if the limit is too low, the extra workers will only end up waiting in line. For deployments with very large volumes, switching to a faster or batched LLM endpoint, or moving the OCR step to a GPU based engine, would give a further speedup. These changes can be applied gradually as the queue waiting time grows, without rewriting any code.

These recommendations build on how the current system already handles résumés. When an applicant submits a résumé, the system does not process it right away. The upload is saved and a job is added to a background queue, so the applicant gets a response immediately. Two background workers then take over: the OCR worker reads the text from the PDF, and the AI worker scores it against the job profile. Each worker can handle three résumés at the same time. The OCR worker uses the CPU heavily because reading text from images is a slow operation, while the AI worker mostly waits for the OpenAI API to reply, so it does not use much CPU. Running both at the same concurrency keeps them in step with each other, which means résumés do not pile up at either stage.
